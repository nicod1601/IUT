       public class Test1

   }

  Travail à Faire
1  Compiler, exécuter

Par défaut les valeurs sont mises à null ou à 0 dans le constructeur, bien que celui-ci ne fasse rien. Nous verrons ultérieurement que nous pourrons nous passer de ce constructeur sous certaines conditions.
Les attributs sont initialisés avec une valeur par défaut :
  •  0 pour les entiers, caractères et réels
  •  false pour les booléens.
   
  •  null pour les types "objet".

Nous pouvons imaginer à partir du TP précédent, une classe ne possédant que des membres (attributs et méthodes) de classe.

Une telle classe est alors qualifier de classe utilitaire.
Elle offre un ensemble de méthodes autonomes dans un environnement stable (qui n'évolue pas).

par exemple pour utiliser la méthode sqrt de la classe Math, qui prend en paramètre un réel, nous n'avons pas besoin d'instancier un Objet Math dans un état donné, pour obtenir un résultat.

 Une classe utilitaire, n'a pas besoin d'être instancier pour être utilisée. Ce serait même une hérésie de vouloir instancier une telle classe.

Nous vous proposons pour tester cela les deux classes suivantes :

public class Util
{
   public static int min (int val1, int val2 )
   {
      if ( val1 < val2 ) return val1;
      return val2;
   }
   public static int max (int val1, int val2 )
   {
      if ( val1 > val2 ) return val1;
      return val2;
   }
}

public class TestUtil
{
   public static void main(String[] a)
   {
      Util util = new Util();

      System.out.println ( Util.min ( 12, 4 ) );

   }

- un attribut d'instance numInstance : numéro d'ordre d'instanciation de l'objet courant.

La gestion des valeurs de ces attributs pourra se faire au niveau de chaque constructeur.

Nous vous proposons également de modifier la méthode toString() comme suit :
public String toString()
{
   return "(" + xxx.numInstance + "/" + xxx.nbInstance +") [" + xxx.forme + " : " + xxx.couleur + " (" + xxx.taille +" cm) ]";
}

Attention il faudra remplacer les xxx.

Ce qui doit nous donner, si nous relançons le programme Test2, la trace d'exécution suivante :

Exercice 6 : la méthode toString() retour haut de page

Le but de la méthode toString() est de retourner une représentation textuelle d'un objet. Nous pourrions très bien utiliser un autre nom, mais Java s'appuie sur ce nom de méthode pour effectuer certains traitements.
Nous allons étudier le comportement de java et de cette méthode à travers un exemple:

Reprenons encore et toujours notre classe Piece (munie du constructeur sans paramètre et du contructeur paramétré). Ecrivons la classe Test6 suivante :

public class Test6
{
   public static void main (String[] arg)
   {
      Piece p1 = new Piece();
      System.out.println ( p1.toString() );
      System.out.println ( p1 );
   }
}

  Travail à Faire
6.1  Que remarquons-nous? que pouvons-nous en conclure ?
6.2  Enlever, la méthode toString() de la classe Piece, compiler, exécuter, conclure

Il existe un réel avantage de ne pas mettre l'appel explicite à la méthode toString.
Voici une illustration :
public class Test6Bis
{
   public static void main (String[] arg)
   {
      Piece p1 = null, p2=null;

      p1 = new Piece();

      System.out.println ( p1 );
      System.out.println ( p1.toString() );
      
      System.out.println ( p2 );
      System.out.println ( p2.toString() );
      
   }

}
Mais est-ce un réel avantage, n'est-ce pas un risque de manipuler des objets null.

7.2  Compiler, et vérifier la réponse à la question précédente.
7.3  Que faudrait-il faire pour comparer le contenu de ces deux objets.

Avant d'écrire le code de la méthode spécifique , nous devons définir quel est le critère d'égalité.
Nous pourrions en effet dire que deux objets de type Piece sont identiques s'ils ont la même forme ou la même couleur ou la même taille ou le même numéro d'instance ou toute autre association de ces quatre caractéristiques.
Il parait évident que si nous tenons compte du numéro d'instance dans la comparaison du contenu de deux objets, nous n'aurons jamais deux objets identiques, à moins que ce soit le même objet.
Donc nous ne sommes pas obligés de prendre en compte tous les attributs pour comparer le contenus de deux objets, tout dépend des spécificités de notre domaine d'étude.

   Pour cet exercice nous ne retiendrons que les critères de forme et de couleur
Nous verrons peut-être ultérieurement à appliquer d'autres critères.

Exercice 8 : la méthode retour haut de page

Nous souhaitons en plus de savoir si une Piece est différente d'une autre, savoir si elle inférieure ou supérieure à une autre Piece. Nous disposons pour cela de la méthode qui permet de comparer deux instances entre elle, tout comme les opérateurs <, <=, > et >= comparent des nombres.

Lorsque la méthode retourne une valeur négative cela signifie que l'objet courant est plus petit que l'objet passé en paramètre.
Lorsque la méthode retourne une valeur positive cela signifie que l'objet courant est plus grand que l'objet passé en paramètre.
Lorsque la méthode retourne la valeur 0 cela signifie que les deux objets (courants et en paramètre) ont le même contenu.

Attention la notion d'ordre entre les instances est à définir selon certains critères (attributs) comme nous avons pu le faire avec la méthode .
Si rien n'empêche les méthodes et d'avoir des critères différents, Java Préconise vivement qu'il y ait une cohérence entre ces deux méthodes.

Exercice 9 : exemple de mise en application retour haut de page

Soit la classe Piece (incomplète) suivante :

public class Piece implements Comparable<Piece>
{
   private static int cptInstance;

   private String forme;
   private String couleur;
   private int    taille;
   private int    numInstance;

   public String toString()
   {
      return "(" + numInstance +"/" + cptInstance +") [" + forme + " : " + couleur + " (" + taille +" cm) ]";
   }

   public Piece()
   {
      this("Sphere", "Bleue", 4 );
   }

   public Piece(String forme, String couleur, int taille)
   {
      this.forme   = forme;
      this.couleur = couleur;
      this.taille  = taille;

      numInstance = ++cptInstance;
   }

}


Pour le moment nous n'entrerons pas dans le détail de l'instruction : implements Comparable<Piece>
Retenons uniquement que cela nous oblige à définir une méthode int compareTo(Piece p) dans la classe Piece.
